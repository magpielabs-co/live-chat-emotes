# Build instructions

How to reproduce the Firefox bundle from this source archive. This document
is aimed at AMO reviewers — it should let you go from this zip to the same
`dist-firefox.zip` that was submitted alongside it.

## Prerequisites

- Node.js 20.x (LTS) or newer
- npm 10.x (ships with Node 20) or newer

No global tools required.

## Build steps

From the extracted source archive:

```
npm install
npm run build:firefox
```

That produces:

- `dist-firefox/` — the unpacked extension, loadable as a Temporary Add-on
  at `about:debugging` for local verification
- `dist-firefox.zip` — the zipped package matching the AMO submission

## What the build does

The pipeline is two stages.

1. `npm run build` runs `vite build` (via `@crxjs/vite-plugin`). This
   transpiles the TypeScript sources under `src/` into a Chrome MV3
   bundle written to `dist/`.

2. `node scripts/build-firefox.mjs` then derives the Firefox build:
   - Copies `dist/` to `dist-firefox/`.
   - Patches `dist-firefox/manifest.json`:
     - Injects `browser_specific_settings.gecko` with the extension ID
       `live-chat-emotes@magpielabs.co` and `strict_min_version: 121.0`.
     - Rewrites `background.service_worker` to `background.scripts`.
       Firefox MV3 currently disables `background.service_worker` and
       rejects it during temporary add-on install, so we use the
       event-page form instead. The background code is pure
       `chrome.*` event listeners with no SW-only APIs (no `FetchEvent`,
       no `clients`, no `skipWaiting`), so the two forms are
       behaviourally equivalent.
     - Strips any Chrome-only dev-time fields (e.g. `key`).
   - Zips `dist-firefox/` into `dist-firefox.zip`.

The source manifest in `manifest.config.ts` is intentionally Chrome-clean;
the Firefox-specific transforms live only in
`scripts/build-firefox.mjs` so the Chrome and Firefox builds stay in
lockstep without drift.

## Source layout

- `src/background/index.ts` — background event-page script. Listens for
  `chrome.runtime` and `chrome.storage` events, fetches 7TV emote sets
  on install and on demand, broadcasts catalog updates to content scripts.
- `src/content/` — content scripts injected into the YouTube live_chat
  iframe (chat renderer, autocomplete, emote picker, mutation observers).
- `src/popup/` — extension popup UI (settings: emote set ID, emote size,
  enable/disable autocomplete and rendering).
- `src/lib/` — shared modules (7TV API client with URL allowlisting,
  storage helpers, message types, shared types).
- `tests/` — vitest unit tests for pure logic.
- `scripts/build-firefox.mjs` — the post-build script described above.
- `manifest.config.ts` — MV3 manifest definition consumed by
  `@crxjs/vite-plugin`.
- `vite.config.ts` — Vite + crxjs configuration.

## No remote code

This extension does not fetch or execute any remote JavaScript. All
executable code is in this source archive.

The extension makes HTTPS requests to two 7TV endpoints:

- `https://7tv.io/v3/emote-sets/<id>` — emote set metadata (JSON)
- `https://cdn.7tv.app/emote/<emote-id>/...` — emote image assets
  (WEBP/AVIF/PNG/GIF)

Both hosts are allowlisted in `src/lib/seventv.ts::safeHostUrl()` to
prevent a compromised or spoofed API response from injecting arbitrary
URLs into `img.src`. The fetch is constrained with `AbortController`
(10s timeout), a 10 MB Content-Length cap, content-type checking, and
`credentials: 'omit'`.

## What's not in this archive

Build outputs (`dist/`, `dist-firefox/`, `*.zip`), dependencies
(`node_modules/`), VCS metadata, and IDE state are excluded. They must
be regenerated by running `npm install` and `npm run build:firefox`
against this source.
